# T17 — Neuron/Expert Mapping Strategy: v3 → v4 → v5 (targeted variants)

**Status:** draft / open questions
**Author:** working notes, 2026-05-14
**Scope:** Gemma 4 26B-A4B MoE (30 layers × 128 experts × top-8 routing).
**Goal:** define a defensible v5 recipe that produces three specialist 98-experts variants — `v5-code`, `v5-science`, `v5-math` — each beating the relevant subset of v4/128e on its target benchmarks while staying within 2pp of 128e on the off-target benches.

---

## 0. Why a new method is needed

v4 lost **−4.04pp on GPQA Diamond** (73.23% → 69.19%), concentrated in **Chemistry**. The single most likely cause is that the signal that decided which 30 experts/layer to drop **had no chemistry questions, no GPQA-style multi-step domain reasoning, and only 128 free-generated tokens of activation per prompt**. Specialist chemistry experts ranked low and were dropped without ever being asked a chemistry question. The same vulnerability exists for code (LCB-medium −5.45pp) and math (AIME −6.67pp). The v5 method is designed to fix this class of problem by using **the actual benchmark's PASS traces from 128e** as the primary signal, and downgrading the synthetic 40-prompt set to a "general competence" prior.

## 1. History — v3 (single-class teacher-force, p16)

### Inputs
- `scripts/per_question_teacher_force.json` — 40 prompts (8 per class × 5 classes: math/logic/code/science/creative), each prompt teacher-forced against the **128e model's own greedy continuation at 128 new tokens**. Same prompt set as v4; the difference is in *what is recorded*, not what is fed.
- Per-expert tracker columns: `wnorm` (norm of weighted expert output), `rnorm` (raw norm before router weight), `wsum` (mean router weight), `tc` (token count routed), `cc` (call count).

### Scoring
```
score[layer][expert] = Σ_q (wnorm_q × α + tc_q),  α = 2.0
```
Pooled across all 40 questions. **Single number per (layer, expert).**

### Drop rule
- Drop bottom-30 per layer (→ 98e) or bottom-19 (→ 109e).
- `protect_top = 16`: the 16 highest-scoring experts per layer are **immune** from dropping even if they fall low elsewhere. This guarded against overaggressive ranking instability.

### Result
- 109e v3 → 77.78% GPQA-D (ties 128e + 2pp lift on some subdomains).
- 98e v3 → 75.25% GPQA-D (= 128e parity at 23% fewer experts).
- LCB-medium v3 → not measured at parity.

### Why it worked
The 40-prompt set, while synthetic, was **uniformly distributed across 5 broad domains** and `wnorm` + `tc` together capture both *how strongly an expert fired* and *how often the router picked it*. The pooled score is a "broad utility" measure. Protect-top-16 prevents the bottom of the rank list from dropping experts that *are* important for some narrow case (insurance against the signal's coverage gaps).

### Failure mode
Pooling treats a math-only specialist as low-utility (it only fires on 8/40 prompts). On a benchmark whose competence depends on that specialist (e.g. AIME), the pooled score is exactly wrong. v3 partially recovered this via `protect_top`, but only for the top 16 — anything ranked 17–30 across pooled is dropped.

## 2. History — v4 (multi-class CD-map, max-over-normalized-classes)

### Hypothesis
A class-specific specialist (an expert that fires only on math) gets ranked low by pooled scoring even when its peak class-specific contribution is huge. If we score **per class** and aggregate via **max**, the specialist gets rescued: it tops the math class even if it bottoms creative.

### Inputs
- `scripts/expert_neuron_v4.json` — same 40 prompts, but the tracker is **kept partitioned by class** (5 classes × 30 layers × 128 experts). Each entry has the same five tracker columns plus a `neuron_act[704]` array (per-neuron squared activation sum, unused by the current drop-map generator but reserved for later residual-expert work).
- Generated by `scripts/expert_neuron_analysis_v4.py`:
  - **Greedy generation, 128 new tokens, on 128e itself** — the model's own continuation of each prompt.
  - The continuation is then **replayed with forward hooks** to capture per-expert activation statistics.
  - **No correctness filter** — even if 128e generated a wrong answer, that path's experts still contribute to the signal.

### Scoring (`scripts/generate_drop_map_multiclass.py`)
```
1. base[class, layer, expert] = wnorm × 2.0 + tc        # same α as v3
2. norm[class, layer, expert] = base / mean(base[class, layer, :])
   (per-class, per-layer unit-mean normalization — cross-class scales fair)
3. score[layer, expert] = max_class norm[class, layer, expert]
   (alternatives: mean / geomean / sum — published v4 uses max)
4. order = argsort(-score[layer]) ; protected = order[:16]
5. drop = sorted(non-protected, ascending)[:30]
```
Output JSON is drop-in compatible with v3: `{layer_str: [list-of-30-dropped-expert-ids]}`.

### Result
- 98e v4 → measured comprehensively in the 2026-05-13/14 vLLM suite:
  - ARC −0.26pp, GSM8K −2.00pp, MATH500 tie, AIME **−6.67pp**, IFEval −7.00pp, HE −2.44pp, HE+ **+1.22pp** (v4 wins), LCB-medium −5.45pp, GPQA-D **−4.04pp (Chemistry concentrated)**.
- Geomean variant built but never deployed; "deeper-aggregate-score doesn't transfer to generation" pathology, same family as v6I (see `memory/feedback_optimizer_off_manifold.md`).

### Why it underperformed v3 on GPQA-D Chemistry
1. **Signal had zero chemistry items.** The "science" class has 8 prompts: quantum entanglement, CRISPR, fusion, Higgs, mRNA, dark matter, plate tectonics, antibiotic resistance. None are graduate-level chemistry, none require the multi-step domain reasoning GPQA-D demands.
2. **Continuations were 128 tokens of 128e's own greedy output.** Most domain-specialist circuits engage much later in a real CoT (after the problem is parsed and a strategy chosen). 128 tokens captures the *intro paragraph*, not the *reasoning kernel*.
3. **No correctness filter.** If 128e lazily answered a chemistry question wrong in 128 tokens, the experts that "produced the wrong answer" still got top scores for the science class. Optimizing for "what fires on the wrong answer" is anti-correlated with what we want.
4. **Per-class unit-mean normalization is fair across classes but blind to class-difficulty.** Math experts and creative experts get equal voice in the max even though the benchmarks reward them very differently.

The Stop-hook lesson from this session applies in reverse: **NOTHING IS PRE-EXISTING. The v4 regression is owned by v4's recipe, not by "GPQA being noisy."**

## 3. v5 — targeted-variant strategy

### Core idea
Split the drop signal into **two priority tiers** and **one targeted PASS-trace channel per variant**:

| tier | source | weight | role |
|---|---|---|---|
| **Tier-A (generic base)** | the existing 5-class synthetic prompts (math/logic/code/science/creative), 128 tokens, greedy on 128e | `w_generic = 1.0` | base competence floor; prevents tunnel-vision |
| **Tier-B (targeted PASS)** | the 128e-PASS trajectories from the variant's target benchmark(s), full-length CoT | `w_targeted = 4.0–10.0` (TBD per question 8) | specialization signal that rescues the right experts |

The variant is what the Tier-B target is:

| variant | Tier-B target benches | Tier-A weight | Tier-B weight |
|---|---|---|---|
| **v5-code** | HumanEval-164 (163 PASS), HumanEval+ (148 PASS), LCB-medium-55 (49 PASS), MBPP-500 (pending) | 1.0 | TBD |
| **v5-science** | GPQA-Diamond (145 PASS), ARC-Challenge (~1043 PASS from acc≈89%, sampled subset OK) | 1.0 | TBD |
| **v5-math** | GSM8K-100 (95 PASS), MATH500-100 (92 PASS), AIME24-30 (22 PASS), AIME25 (TBD — not run yet) | 1.0 | TBD |

### Aggregation rule (proposed)
For each variant, build a **6-class score grid** = 5 generic classes + 1 (or more) targeted-PASS classes. Then:

```
score[layer, expert] = max(
    w_generic × max_{c ∈ generic}  norm[c, layer, expert],     # broad floor
    w_targeted × max_{c ∈ targeted} norm[c, layer, expert]     # specialization peak
)
```

This is the **weighted-max** generalization of v4's max. With `w_targeted ≫ w_generic`, the variant's PASS specialists are guaranteed to outrank generic-only experts at the bottom of the rank list — but generic competence is *not* dropped, just outranked when there's a conflict at the drop boundary.

Alternative formulations to investigate (see open question 3):
- **Channel-replacement**: drop one generic class (e.g. "math" generic for v5-math), use only the targeted PASS for that domain → simpler but loses the synthetic prior's noise-floor benefit.
- **Channel-addition**: keep all 5 generic + add 1 targeted → what is sketched above.
- **Hierarchical**: protect top-K by targeted score, then bottom-drop by generic. Closer to v3's `protect_top` philosophy.

## 4. Open questions

These are the design decisions where the right answer isn't obvious from first principles. Each one has a **proposed default** to unblock progress and a **shape of the experiment** that would resolve it.

### Q1 — How many tokens of activation per prompt is enough?

**Current:** 128 new tokens, greedy on 128e (v3/v4 both).
**Why it might be too short:**
- Domain-specialist circuits in Gemma 4 (per analysis_v5.py exploration) appear to engage **after the problem-restatement preamble** (~50–100 tokens) and during the **mid-CoT reasoning step** (~300–1500 tokens).
- A 128-token continuation captures preamble + maybe one reasoning step. That favors *parsing experts* and *style experts* over *domain experts*.
- For GPQA-D in particular, the 128e canonical eval uses `thinking_token_budget=12288` and average completion ≈ 6–8k tokens. The signal-capture run never sees ≥98% of what the model actually does at eval time.

**Why we capped at 128:**
- v4 analysis took ~9h on CPU fp16 at 128 tokens × 40 prompts. Going to 1024 → ~70h. Going to 8192 → ~560h. **CPU-only is the bottleneck**, not the algorithm.

**Investigation:**
1. Pick **3 representative prompts** (one math, one code, one science). Run analysis at `{128, 512, 2048, 8192}` tokens. Plot per-expert score-rank as a function of generation length per layer.
2. Stability metric: **Kendall-τ** between consecutive lengths on the bottom-30-rank set per layer. Where the τ plateaus → that's the safe minimum. Hypothesis: 128 → 512 changes the ranking significantly; 512 → 2048 stabilizes; 2048 → 8192 marginal.
3. **GPU**: move analysis to GPU (RTX 3090, bf16 with a tiny attention chunk size) to make 2048-token analysis tractable (~6–12h instead of weeks).

**Proposed default for v5:**
- **Tier-A (generic synthetic)**: keep 128 tokens. The point of Tier-A is broad competence, not specialization depth. 128 captures the prior fine.
- **Tier-B (PASS traces)**: use the **full saved CoT** from the canonical eval samples_*.jsonl (typically 1–8k tokens) — these are *already-generated*, so we pay zero generation cost. We only pay the replay-with-hooks cost. **This makes the answer to Q1 "as long as the actual completion was."**

This is the single biggest free lunch in the v5 design: the PASS traces *already exist on disk*, so the signal can be 10–50× longer than v4's at no generation cost.

### Q2 — Which tests, and is current coverage sufficient?

**Decision matrix per variant:**

| variant | required Tier-B benches | enough? | action if not |
|---|---|---|---|
| v5-code | HumanEval(+) + LCB-medium | **yes** (360 PASS, long-CoT) | optional augment with MBPP-500 if 128e run completes in time |
| v5-science | GPQA-D + ARC-Challenge | **yes** for GPQA; ARC is mostly short MCQ (less signal-rich per question) | subsample 200 ARC PASS to keep ratio sane; consider running **MMLU-Pro STEM subset** for breadth |
| v5-math | GSM8K + MATH500 + AIME24 | **borderline** — only 22 AIME24 PASS, and AIME is the hardest target | run **AIME25 (30q)** on 128e NVFP4A16 (~1.5h); consider Minerva-MATH-500 full 500q (we currently only have a 100-subset) |

**Question — "which 128e-passes-but-98e-fails to use?"**
This is sharper than just using all 128e PASS. The genuinely informative samples are:

- **Set A**: 128e PASS ∧ v4 PASS — "easy enough that any v4-style prune is fine"; lower marginal value.
- **Set B**: 128e PASS ∧ v4 FAIL — **the regression set**; these are the questions v4 broke. Maximally informative.
- **Set C**: 128e PASS only (no v4 measurement) — neutral but usable.

**Proposed strategy:** weight Tier-B samples by class:
- Set B: `weight = 3.0` (the regression — what we're trying to fix)
- Set A: `weight = 1.0` (positive control — confirm we keep these)
- Set C: `weight = 1.0` (when v4 measurement unavailable)

**Measured Set-B sizes (128e PASS ∧ v4 FAIL)** — to be computed by the analysis script before each variant build. Quick reference from the per-bench v4 deltas:
- HE: 5 problems (99.4 → 96.95 = 4q delta; ~5 in the regression set)
- HE+: 0 problems (v4 actually wins) — but the Set-A "preserve" pool is large.
- LCB-medium: 3–4 problems regression (49 → ~46)
- GSM8K: 2 problems (95 → 93)
- MATH500: 0 (tie)
- AIME24: 2 (22 → 20)
- GPQA-D: 8 problems (145 → 137 estimated from −4.04pp on 198)

For v5-science the **8 GPQA regression questions** are the highest-information signal in the entire corpus — every replay-with-hooks pass on those is worth ~10× a generic prompt.

### Q3 — Add or replace the generic class with the targeted one?

**Three concrete options:**

| option | description | pro | con |
|---|---|---|---|
| **A. Add** | 5 generic + 1 (or 2) targeted; weighted-max | preserves broad floor; symmetric across variants | targeted signal can be drowned out if `w_targeted` too low; tunable |
| **B. Replace** | for v5-code, swap the "code" generic class for "code_targeted"; keep math/science/logic/creative generic | one fewer knob; cleaner | loses the synthetic-code prior; if PASS traces are biased (e.g. all HE is Python), generic Python prompts can no longer compensate |
| **C. Add + protect-by-targeted** | 5 generic + 1 targeted; `protect_top` is computed against **targeted score only** | every variant guarantees its specialists are kept; generic ranking decides the bottom | very strong specialization; risk of overfitting to PASS-trace style |

**Proposed default:** **Option A** with `w_generic = 1.0, w_targeted = 4.0`. The 4× ratio gives the PASS class a clear voice without nuking the generic prior. Option A is also the **most reversible** — if v5-code disappoints on, say, MBPP, we just retune the weight, no rebuild of the analysis JSON. Options B and C should be ablated **only if Option A passes the canary** (HE > 97%, LCB ≥ 89%, on-target lift ≥ +1pp).

### Q4 — Same token gen limit for generic and targeted?

**Proposed:** **No — and not for the reason you might think.**

- **Tier-A generic stays at 128 tokens, freshly generated.** Its purpose is "broad activation prior under controlled conditions." 128 tokens is *intentionally short* to keep the prior diffuse — we don't want generic competence to look specialized.
- **Tier-B targeted uses the full existing PASS-trace length** (no new generation). Average lengths from the 128e canonical run:
  - HumanEval: ~150–400 tokens (short, no reasoning budget activated typically)
  - LCB-medium: ~2k–10k tokens (heavy CoT)
  - GPQA-D: ~6k–10k tokens (reasoning budget hits the cap)
  - MATH500: ~1k–5k tokens
  - AIME24: ~6k–12k tokens

Because Tier-B is *replay-only* (no generation), longer is essentially free. The only cost is forward-pass time on solidPC's 3090, which at bf16 batch-1 with chunked attention is ~2–5 tokens/sec per replay → a 10k-token replay is 30–80 min, **× 145 PASS GPQA-D items = 75–195 hours**.

That's the catch: **full-length replay of all PASS traces is too expensive** to do naively. Mitigations:

1. **Subsample**: pick 30–50 PASS traces per bench, weighted by Set-B preference (the regression set).
2. **Sliding-window replay**: instead of replaying the entire 10k-token sequence in one forward pass (KV cache + memory), replay in 2k-token windows with overlap. Each window contributes independently to the tracker.
3. **GPU bf16**: a single 2k-token forward pass on Gemma 4 26B-A4B is ~5–10s on a 3090. 145 PASS × 5 windows × 8s = ~1.6h per bench. **Tractable.**

### Q5 — MoE router quality: silent-empty and rumination

This is the **most important architectural question** in v5. We have direct evidence (task #207) that v4 has a *silent-empty-output failure mode*: gsm8k 5/100, HE 1/164, HE+ 1/164 problems where the first generated token is `<eos>` or equivalent. Concurrently, v4 on LCB-medium without `thinking_token_budget` *ruminates*: 30–50k chars, 99 duplicate ```python``` fences, finish_reason=length.

**Hypotheses:**

1. **H1 — Router top-k saturation on pruned experts.** v4 has 98/128 experts. The router was trained on 128. After dropping 30, the top-8 routing for some prompts now selects experts that *were not top-8 in 128e* — the router falls through to its second/third-choice activations. If those secondary experts are mis-calibrated for the prompt class, the output collapses to a single token or loops.
2. **H2 — Router-weight redistribution failure.** The softmax over surviving experts is not re-normalized. After dropping 30, the kept-8 router weights still sum to <1 in cases where the original top-8 included a dropped expert. The downstream consumer sees a *partial* expert mixture → degenerate behavior.
3. **H3 — KV-cache routing drift.** During long-CoT generation, the routing distribution drifts further from training as the conversation length grows. Past a critical token count, no surviving expert is well-suited → rumination loop.

**Investigations:**

1. **Direct measurement**: instrument vLLM (or use a small HF replay) on the 5+1+1 = 7 v4-failing problems. Log per-layer top-8 router output **with and without** the 30 dropped experts. Compute:
   - Probability mass on dropped experts (should be 0 in v4 since they're absent — but ask what the router *would* have chosen).
   - Entropy of the survivor distribution vs the 128e baseline at the same token position.
   - First-token-after-decode-anomaly: where in generation does the route diverge?
2. **Router recalibration**: 2–10k tokens of calibration data through the surviving 98 experts to re-fit the gating layer (REAP-style). Apply to v4 and re-measure GPQA-D. Cheap (1–2h) and **independent of v5 design**.
3. **Top-k re-bound**: when ≥3 of original top-8 are dropped at a given layer, force a fallback to top-12 over survivors for that token. Implementation lives in `expert_drop.py`'s GGUF builder or, more cleanly, a llama.cpp side patch. **Risk: high implementation cost, may break llama.cpp upstream sync.**

**Decision:** **router recalibration is task #207's fix and is a prerequisite for trusting any v5 score.** v5 should be built *and then re-evaluated with and without the recalibrated router*; if the +1pp targeted lift comes entirely from router fix, the v5 method is moot for now.

A separate task track (`T17.R — router probe`) should run in parallel with v5-code build.

### Q6 (newly raised) — Should the per-neuron data finally be used?

v3 and v4 both saved `neuron_act[704]` per expert but the drop-map generators *ignore it*. The original purpose was the abandoned "residual expert" branch (REAP Theorem 1 rejection). Now that we're targeting specialization, **per-neuron data is a natural drop-granularity refinement**: instead of "drop expert E in layer L," do "drop the lowest-25% neurons of every expert kept in layer L." This is a parallel research track to v5 expert-drop and could be combined with it (v5-code expert-drop + 25% per-expert neuron-prune). **Out of scope for the first v5-code build; document as v5.5.**

## 5. Concrete v5-code build plan

### Step 1 — Targeted-PASS extractor (`scripts/extract_pass_traces.py`)
New script.
- Reads `samples_*.jsonl` files from `eval_results_vllm_suite/128e/{humaneval_full,humanevalplus_full,lcb_medium_55}/.../lm_eval_out/`.
- Emits `scripts/v5_code_pass_traces.json`:
  ```json
  {
    "metadata": {"variant":"v5-code", "n_traces":360, "set_a":?, "set_b":?, "set_c":?},
    "traces": [
      {"bench":"humaneval","doc_id":0,"set":"A","prompt":"...","completion":"...","tokens":[...]},
      ...
    ]
  }
  ```
- Filter rule: row's flexible-extract / pass@1 = 1.0 on 128e. Set classification needs the v4 samples too.
- Token budget per trace: full completion, capped at 10k tokens (sanity).

### Step 2 — Mapping run (`scripts/expert_neuron_analysis_v5.py`)
- Loads 128e on **GPU bf16** (RTX 3090, chunked attention). v4's CPU fp16 was a workaround; for v5 we accept the GPU cost in exchange for tractable longer signal.
- **Tier-A**: rerun the 40 synthetic prompts at 128 tokens (existing logic). Save under `categories.generic_<class>` keys.
- **Tier-B**: for each PASS trace, tokenize the full `prompt + completion`, replay-with-hooks. Save under `categories.targeted_<bench>` keys.
- Save windowed if any trace > 4k tokens.
- Output: `scripts/expert_neuron_v5_code.json` (likely ~600–900 MB, well within disk budget).
- Estimated runtime on 3090: ~12–18h.

### Step 3 — Drop-map (`scripts/generate_drop_map_v5.py`)
- Extension of `generate_drop_map_multiclass.py`.
- New CLI: `--variant {code,science,math}`, `--w-generic 1.0`, `--w-targeted 4.0`, `--set-b-multiplier 3.0`.
- Aggregation: weighted-max (Option A from Q3).
- Same protect_top=16, drop=30.
- Output: `scripts/v5_code_drop_map.json` + sidecar summary.

### Step 4 — Build pipeline (`scripts/build_98e_v5_code.sh`)
Verbatim from `build_98e_cd.sh`, with:
- `DROP_MAP="scripts/v5_code_drop_map.json"`
- `SUFFIX="-v5-code"`
- Outputs `google/gemma-4-A4B-98e-v5-code/` + `gemma-4-A4B-98e-v5-code-Q6_K.gguf` + NVFP4A16 quant.

### Step 5 — Canary + full eval
1. **Smoke** (1h): 10q HE + 10q LCB + 10q GPQA → instant GO/NO-GO.
2. **Full vLLM suite** via existing `eval_suite_chain.sh` with v5-code as a new variant. ~10h.
3. **Comparison table** vs 128e + v4 on all 9 benches; decision gate below.

### Decision gate for v5-code
| outcome | criterion | action |
|---|---|---|
| **PASS** | HE+ ≥ 98% AND LCB ≥ 91% AND no off-target bench drops > 3pp vs 128e | publish to HF as `gemma-4-A4B-98e-v5-code-it`; proceed to v5-science |
| **MARGINAL** | HE+ 95–98% AND LCB 88–91% | run weighted ablation (`w_targeted ∈ {2,4,8,16}`); pick best |
| **FAIL** | any code bench worse than v4 | document; revisit Q3 (Replace vs Add) and Q5 (router) |

## 6. Risks

- **R1: Set-B over-fitting.** Weighting the regression set 3× could push v5-code to *exactly* fix v4's HE/LCB failures while breaking new ones. Mitigate by holding out a stratified 10% of Set-B from the analysis input and verifying those still pass post-prune.
- **R2: GPU memory under bf16 replay.** Gemma 4 26B-A4B at bf16 with 8k-token KV cache won't fit single-3090 24GB. Mitigate with chunked-attention (2k window, overlap 256) or `device_map="auto"` with offload to CPU for non-active layers.
- **R3: Tier-B trace contamination from chat template.** If the saved completions include the `<bos>...<eos>` chat scaffolding, the replay's first ~30 tokens fire stylistic experts, not domain. Strip scaffolding before replay.
- **R4: Router pathology dominates.** If Q5 turns out to be the real bottleneck, v5 specialization gains are bounded by ~+0–1pp. The v5 method is still useful for *which experts to keep*, but the bigger win is router recalibration. **Track Q5 in parallel.**
- **R5: AIME data thinness for v5-math.** Only 22 PASS traces at competition-level math. v5-math may need to wait until AIME25 + MMLU-Pro STEM are run on 128e NVFP4A16.
