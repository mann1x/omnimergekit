# Caliber Learnings

Accumulated patterns and anti-patterns from development sessions.
Auto-managed by [caliber](https://github.com/caliber-ai-org/ai-setup) — do not edit manually.

- **[pattern]** To extract HE/MBPP scores from lm-eval JSON, parse `results_*.json` for `pass@1,create_test` (HE) and `pass_at_1,none` (MBPP); LCB lives in a separate `lcb_results.json` with key `pass_at_1`.
- **[gotcha]** A HuggingFace API 401 response typically means the repo was deleted, NOT that auth is missing. Don't waste time trying alternate path casings or re-authenticating — verify with `hf download <repo> config.json` which returns the explicit `RepositoryNotFoundError: 404`. The user namespace itself returning 200 confirms the user exists but the model is gone.
- **[pattern]** When making changes to `omnimergekit.py`, edit the canonical copy at `$WS/scripts/omnimergekit.py` then `cp` it to `/shared/dev/omnimergekit/` for the git repo. Recipe scripts go to both `$WS/scripts/` (executed) and `recipes/microcoder_4b/` (versioned). The repo is the publication artifact; `$WS/scripts/` is the runtime location.
- **[pattern]** After editing `omnimergekit.py`, syntax-check with `$PYBIN -c "import ast; ast.parse(open(...).read())"` and verify CLI flags via `--help | grep -A 6 <flag>` before launching merges (a merge run takes ~10 min on 4B; catching a typo earlier saves the round trip).
- **[pattern]** Long merge+eval runs use a `Monitor` tool with `tail -F` + `grep --line-buffered -E "<phase markers>|pass@1|exact_match|ALL DONE|^\[!]"` on the master log. Match phase headers (`Phase [0-9]`, `ext-eval`), score lines (`pass@1`, `pass_at_1`, `exact_match`), and the failure marker `^\[!]` so a crash is never silent.
- **[convention]** New merge variants must be backwards compatible — every existing recipe (v1b, v2a-h) should reproduce byte-identically when the new flag is omitted (default `None` → legacy code path). State this explicitly in the commit message.
- **[gotcha]** When cloning a variant script (`cp v2j.sh v2k.sh`), the `NAME=` variable AND any `M_DIR`/`LOG_MASTER` paths derived from the old name must be renamed BEFORE launch — otherwise the new run writes into the previous variant's merged/ and gguf/ dirs, overwriting artifacts and starting an llama-server on the old GGUF. Always `grep -n "v2[a-z]" newscript.sh` after cloning to catch stragglers.
- **[pattern]** Before launching a new variant, run `pgrep -af "competence_v2|llama-server.*8099|lm_eval" | grep -v grep` to confirm no stray processes from the previous run are still holding the GPU/port. Phase 3 of each script does its own `pkill llama-server.*8099`, but a misnamed clone can leave the previous variant's server alive on a different model file.
- **[pattern]** Pair every long-running `Monitor` (`tail -F | grep`) with a `ScheduleWakeup` ~20-25 min out as a safety net — `tail -F` on a log path that doesn't yet exist plus a phase-marker grep can silently miss `ALL DONE` if buffering or a crash interrupts the pipe.
- **[gotcha]** lm-eval 0.5+ rejects `--gen_kwargs` values that contain top-level commas: `_cli/utils.py:key_val_to_dict` splits the string on `,` then expects `=` in every piece, so a list like `until=["\nclass","\ndef",...]` errors with `ValueError: not enough values to unpack (expected 2, got 1)`. Encode `until`/`max_gen_toks`/etc. in the task YAML's `generation_kwargs:` block instead of overriding at the CLI.
- **[gotcha]** lm-eval's `!function` YAML tag is path-based, NOT a Python import: `!function pkg.sub.utils.func` is resolved as the FILE `pkg.sub.utils.py` relative to the YAML's directory (via `--include_path`). To reuse upstream helpers like `lm_eval.tasks.humaneval.utils.pass_at_k`, drop a thin re-export module (e.g. `_he_utils.py: from lm_eval.tasks.humaneval.utils import pass_at_k`) into the include dir and reference it as `!function _he_utils.pass_at_k`.
- **[pattern]** When killing a stuck eval run, `pkill -f eval_suite.sh` does NOT take down the spawned `llama-server` on port 8099 — it survives and holds the GPU. Always follow with `pkill -f "llama-server.*8099"` (or check via `pgrep -af llama-server`) before relaunching, otherwise the new run hits a port-in-use or evaluates against the previous model.
- **[convention]** When a non-trivial fix lands during a session, append an entry to `$WS/.wolf/buglog.json` (`bugs[]` array, `bug-NNN` ids — read `max_id` first) with `error_message`, `root_cause`, `fix`, and `tags`. Also `related_bugs` when chaining fixes (e.g. lm-eval CLI regression → YAML !function fallout).
- **[gotcha]** Gemma 4 IT degenerates on raw `/v1/completions` for code prompts (HE produces infinite `"""` loops, MBPP server-500s on garbled retries). For IT models, route HE/MBPP through `/v1/chat/completions` with `--apply_chat_template` instead of completion-style eval. Mirror upstream `humaneval_instruct.yaml` / `mbpp_instruct.yaml` patterns rather than inventing new prompts.
- **[gotcha]** lm-eval's `gen_prefix:` field translates to an assistant-message prefill on chat-completions backends. llama-server with `--jinja` and a chat template that has `enable_thinking` on by default (Gemma 4 IT) returns HTTP 400 `"Assistant response prefill is incompatible with enable_thinking"`. Drop `gen_prefix` from `*_chat` task YAMLs and let `build_predictions_instruct` / `build_predictions` extract code from the natural ```python fence post-hoc.
- **[gotcha]** When a chat-completions task has no `gen_prefix` to absorb the opening code fence, do NOT include `\n```` in `generation_kwargs.until` — it matches the OPENING ```python fence the model emits and halts generation after one token (raw response = empty, score = 0). Rely on the model's native end-of-turn token (e.g. Gemma `<end_of_turn>`) plus cross-arch sentinels (`<|file_separator|>`, `<|im_end|>`, `</s>`) instead.
- **[convention]** `eval_suite.sh` dispatch routes tasks by name suffix: `*_chat` / `*_instruct` → chat-completions server (`--jinja`, no reasoning flags), `gpqa*` / `*_gpqa_*` → GPQA server (`--jinja --reasoning-format deepseek --reasoning-budget 16384`), everything else → plain `/v1/completions`. Name new tasks with these suffixes to inherit the right server config automatically.
- **[env]** The canonical Python env for omnimergekit work is the `omnimergekit` conda env (`/root/anaconda3/envs/omnimergekit/bin/...` on solidPC, `/opt/conda/envs/omnimergekit/bin/...` on pods). `eval_suite.sh` lm_eval discovery prefers it over the `lightseek` `.venv`. Even when the two envs have identical `lm_eval` / `transformers` versions, use omnimergekit so pods, CI, and collaborators reproduce the same setup.
- **[pattern]** When debugging `pass@1 = 0.0` on a chat eval, inspect the `samples_*.jsonl` first: compare `resps` (raw model output) to `filtered_resps` (post-extract). Empty `resps` → premature `until` match or backend error; `filtered_resps` containing only the prompt docstring → extraction filter found no fenced code block. This discriminates between server/template bugs and prompt/until bugs in one look.
- **[gotcha]** Gemma 4 IT's chat template enables a thinking phase by default. On llama-server with `--jinja` (no `--reasoning off`), the server splits output into `reasoning_content` (thoughts) and `content` (answer). If the model exhausts `max_tokens` while still thinking, `content` is empty — lm-eval's `local-chat-completions` backend reads only `content` → 0% pass@1. Fix server-side with `--reasoning off` (preferred over per-request `chat_template_kwargs.enable_thinking=false`; works transparently for any client). Verify via curl probe: `content` should be populated, `finish_reason=stop`.
- **[gotcha]** Upstream `lm_eval.tasks.humaneval.utils.build_predictions_instruct` is geared to `gen_prefix` mode: it assumes the model emits only the function body and truncates at the prompt's docstring close. When you've dropped `gen_prefix` (e.g. to avoid the enable_thinking conflict) the model re-emits the whole function inside a ```python``` fence, and the upstream filter drops the body → `filtered_resps` contains only the prompt docstring, pass@1 = 0. Use a local `build_predictions_chat` filter that regex-extracts the largest ```python``` block instead. Same pattern for MBPP — upstream `build_predictions` returns prose verbatim if no fences, exec'ing as SyntaxError.
- **[pattern]** Run `llama-server` with a per-eval profile, not a single shared config. Sampler/template/reasoning flags that are right for HE chat (`--jinja --reasoning off`, greedy) are wrong for GPQA (`--reasoning-format deepseek --reasoning-budget 16384`) and wrong for raw-completion code tasks. `eval_suite.sh` exposes `--{plain,chat,gpqa,gsm8k}-server-args` plus base flags (`--ctx`, `--ngl`, `--threads`, `--parallel`, `--server-base-args`, `--server-args`); profiles are also overridable via `PROFILE_PLAIN`/`PROFILE_CHAT`/`PROFILE_GPQA`/`PROFILE_GSM8K` env vars. Each profile restart goes through `stop_server` (TERM→KILL escalation) + `wait_for_vram_clear` so VRAM is reclaimed between profiles.
- **[gotcha]** A llama-server zombie from a prior `eval_suite.sh` run can hold ~22 GB on a 24 GB 3090, causing the next server start to fail with `cudaMalloc failed: out of memory` and a misleading `ERROR: server failed to start` in the eval log. Before launching, check `nvidia-smi --query-compute-apps=pid,process_name,used_memory --format=csv,noheader`; if a stray `llama-server` is listed, `kill -9` it and verify VRAM drops below ~1 GiB before retrying.
- **[pattern]** Every `start_server` invocation should write a header (timestamp, profile name, full cmdline) to the per-profile server log AND echo the cmdline to the eval-suite log — the binding between server output and its launch arguments is essential when reviewing a failed run hours later.
- **[gotcha]** `quantize_gguf.py`'s `auto_ngl()` is intentionally conservative for partial-offload imatrix on a 24 GB 3090 — the prior 0.60×available × ratio×35 layers formula left ~10 GB unused on a 31B F16 model. The current heuristic uses 0.85×available + 50-layer assumption, leaving ~3-4 GB headroom for KV cache / compute buffers. If you tighten it further, watch for `cudaMalloc failed` during `imatrix` (different code path from llama-server) and re-check Gemma-class models specifically (62 layers).
- **[gotcha]** Head-pruning via `prune_local_heal.py` does NOT reduce GGUF file size — the surviving tensor SHAPES are unchanged (heads are zeroed/healed, not sliced), so Q4_K_M of base-31B and `*-he125` are byte-for-byte equal in size (both ~17.40 GB). To actually shrink the file you must rewrite q/k/v/o_proj with reduced rows AND update `config.json` (`num_attention_heads`, `num_key_value_heads`) before GGUF conversion, then re-heal because slicing perturbs the lstsq fit. This was a recurring point of confusion — a smaller model on disk requires structural changes plus a config bump, not just zero-ablation.
- **[gotcha]** Replacing `nn.Linear` submodules on an accelerate-offloaded model (e.g. physical attention head reshape) breaks the offload `weights_map` keyed by original param names, and `save_pretrained` later fails trying to look up replaced parameters. Before swapping submodules: `from accelerate.hooks import remove_hook_from_submodules; remove_hook_from_submodules(model); model.to("cpu"); torch.cuda.empty_cache()`. Then do the surgery, then save.
- **[pattern]** Verify GGUF structural metadata propagated correctly via `gguf-dump --no-tensors <gguf>` (top-level `gemma4.attention.head_count`/`head_count_kv`) AND a `GGUFReader` per-layer-array probe — Gemma 4's `head_count_kv` is a length-60 INT32 array (per-layer sliding vs full-attn KV count), not a scalar. A scalar-only check misses per-layer mismatches. Pattern: read with `r.fields[name].parts[di][0] for di in r.fields[name].data`.
- **[pattern]** Restartable sweep wrappers should skip BOTH `${out}/prune_manifest.json` (prior pass) AND `${out}.broken/prune_manifest.json` (prior canary FAIL output) — otherwise resuming a partly-failed sweep re-runs every band that already produced a verdict. Add the `.broken` skip explicitly; canary failures are still results.
- **[gotcha]** Adding a top-level constant or import via `Edit` near a similar existing line risks a silent duplicate definition (no error, last one wins, but confusing on grep). After multi-line Edits that introduce `SOMETHING = ...` or `import X`, run `grep -n '^<TOKEN>' <file>` to detect duplicates and clean them up before moving on.
